======================================================================
1. RINGKASAN EKSEKUTIF
======================================================================

Dokumen ini menjelaskan rencana pengembangan aplikasi Project Management Planner Gantt agar naik kelas menjadi aplikasi kolaborasi kerja yang lebih profesional, dengan referensi fitur publik dari Cicle Project Management.

Targetnya bukan menyalin Cicle secara mentah, tetapi membangun aplikasi yang memiliki kemampuan sekelas Cicle pada area:

1. Manajemen tugas harian.
2. Monitoring pekerjaan lintas proyek dan departemen.
3. Komunikasi kerja terpusat.
4. Dokumen dan file terstruktur.
5. Aktivitas dan audit trail.
6. Laporan kinerja karyawan.
7. Reminder, notifikasi, dan kalender agenda.
8. Kontrol akses yang aman.
9. Search lintas data.
10. Pengalaman kerja yang sederhana, rapi, dan mudah diadopsi oleh tim Indonesia.

Aplikasi saat ini sudah memiliki kekuatan yang tidak terlalu ditonjolkan oleh Cicle dalam materi publiknya, yaitu:

1. Gantt Chart per project.
2. Department Gantt.
3. Nested subtask.
4. Rollup progress parent task.
5. Working calendar untuk menghitung work days.
6. Actual realization.
7. Lead approval.
8. Multi PIC.
9. Cross-department project involvement.

Strategi pengembangan terbaik adalah mempertahankan kekuatan Planner + Gantt, lalu menambahkan lapisan collaboration operating system seperti Cicle.

Produk akhir yang dituju:

Project Management Planner Gantt menjadi "Department Work Hub", yaitu platform internal perusahaan untuk:

1. Membuat project dan ruang kerja tim.
2. Mengatur pekerjaan dalam board, list, calendar, dan Gantt.
3. Mengelola komunikasi pekerjaan tanpa bergantung pada WhatsApp.
4. Menyimpan dokumen dan link penting.
5. Memantau kinerja individu, departemen, dan proyek.
6. Mengurangi meeting reporting melalui check-in async.
7. Menjamin semua instruksi, komentar, perubahan data, dan persetujuan tercatat.

======================================================================
5. PRINSIP PRODUK DAN ARSITEKTUR
======================================================================

5.1 Prinsip produk

1. Sederhana dipakai oleh karyawan non-teknis.
2. Tetap kuat untuk monitoring manajemen.
3. Semua data penting harus tercatat dan bisa diaudit.
4. Komunikasi kerja harus terhubung dengan objek kerja.
5. Jangan membuat fitur chat menjadi pengganti task; chat hanya mendukung koordinasi.
6. Gantt tetap menjadi pembeda utama.
7. Permission dan audit harus dibangun sebelum fitur kolaborasi terlalu luas.
8. Fitur baru harus incremental dan tidak merusak workflow MVP.

5.2 Prinsip teknis

1. Pertahankan stack React + Vite + Express + PostgreSQL untuk fase awal.
2. Jangan migrasi ke Next.js sebelum kebutuhan SSR, server component, atau fullstack routing benar-benar jelas.
3. Tambah TypeScript secara bertahap, dimulai dari shared type dan service boundary.
4. Gunakan migration SQL eksplisit untuk setiap perubahan schema.
5. Backend harus menjadi pemilik business rule.
6. Frontend hanya mengatur UX dan validasi ringan.
7. Semua input eksternal divalidasi.
8. Semua endpoint mutation dilindungi auth dan permission.
9. Semua perubahan data penting masuk audit log.
10. Hindari dependency baru kecuali jelas meningkatkan kualitas.

5.3 Target arsitektur logical

Layer frontend:

1. pages: route-level screens.
2. components: UI reusable dan feature components.
3. logic/services: API clients.
4. logic/hooks: data fetching hooks.
5. logic/helpers: pure UI/domain helpers.
6. store: UI/session state.

Layer backend:

1. routes: mapping HTTP route.
2. controllers: parsing request dan response.
3. services: business logic.
4. repositories atau query modules: query kompleks yang mulai besar.
5. middlewares: auth, permission, validation, error handling.
6. utils: shared pure helpers.
7. database/migrations: perubahan schema.

Layer database:

1. master data.
2. operational data.
3. collaboration data.
4. audit data.
5. search data.
6. reporting views.

7.2 Middleware auth

Endpoint yang harus publik:

1. POST /api/auth/login.
2. POST /api/auth/accept-invitation.
3. POST /api/auth/request-password-reset.
4. POST /api/auth/reset-password.
5. GET /.

Endpoint lain harus butuh auth.

Header:

1. Authorization: Bearer <token>.

Flow:

1. Middleware membaca Bearer token.
2. Hash token dan lookup ke auth_sessions.
3. Pastikan session belum expired dan belum revoked.
4. Join users.
5. Pastikan user aktif.
6. Set req.user.
7. Controller/service memakai req.user.id dan req.user.role.

2. requireProjectAccess(action).
3. requireObjectAccess(objectType, action).

1. project:create.
2. project:update.
3. project:archive.
4. task:create.
5. task:update.
6. task:move.
7. task:approve.
8. task:delete.
9. comment:create.
10. document:upload.
11. document:archive.
12. announcement:create.
13. schedule:create.

Permission resolution:

1. super_admin selalu allowed.
2. admin allowed untuk semua kecuali action super admin only.
3. project role override global role.
4. object access override project role jika lebih spesifik.
5. creator privilege berlaku untuk object yang dibuat user.
6. backend tidak boleh hanya bergantung pada role dari frontend.

3. Kategori task:
   - selesai.
   - dikerjakan.
   - belum.

   - waiting review.
   - tanpa deadline.
4. Hitung score.
5. Tentukan rating.
6. Generate recommendation.
7. Detect bottlenecks.

Formula detail awal:

1. total_tasks = semua task assigned dalam periode.
2. completed_tasks = status Done atau progress 100.
3. in_progress_tasks = status In Progress.
4. not_started_tasks = status Not Started.
5. overdue_tasks = end_date < today dan progress < 100.
6. no_due_date_tasks = end_date IS NULL.
7. waiting_review_tasks = status Waiting Review.

Rasio:

1. completed_ratio = completed_tasks / total_tasks.
2. overdue_ratio = overdue_tasks / total_tasks.
3. stale_ratio = task in progress lebih dari N hari tanpa update.
4. no_due_date_ratio = no_due_date_tasks / total_tasks.

Score:

1. Base score 60.
2. Tambah sampai 30 poin dari completed_ratio.
3. Tambah sampai 10 poin dari in_progress_ratio sehat.
4. Kurangi sampai 40 poin dari overdue_ratio.
5. Kurangi sampai 15 poin dari stale_ratio.
6. Kurangi sampai 10 poin dari no_due_date_ratio.
7. Clamp 0 sampai 100.

Rating:

1. Bagus: 80 sampai 100.
2. Cukup: 60 sampai 79.
3. Kurang: 0 sampai 59.

Recommendation examples:

1. Jika overdue tinggi: "Prioritaskan penyelesaian task overdue dan review estimasi deadline."
2. Jika not started tinggi: "Pecah task besar menjadi subtask dan mulai task prioritas."
3. Jika waiting review tinggi: "Lead perlu mempercepat approval agar pekerjaan tidak tertahan."
4. Jika no due date tinggi: "Lengkapi deadline agar pekerjaan bisa dipantau."

======================================================================
14. RISIKO DAN MITIGASI
======================================================================

Risiko 1: Scope terlalu besar.

Mitigasi:

1. Gunakan fase.
2. Setiap fase harus punya acceptance criteria.
3. Jangan membangun chat penuh sebelum notification/comment matang.

Risiko 2: Permission terlambat dibangun.

Mitigasi:

1. RBAC fase 1.
2. Semua endpoint mutation wajib auth.
3. Backend selalu validasi izin.

Risiko 3: Activity log tidak konsisten.

Mitigasi:

1. Semua mutation memakai activityService.
2. Buat helper standar.
3. Review endpoint mutation satu per satu.

Risiko 4: Query performance memburuk.

Mitigasi:

1. Tambah index.
2. Pagination.
3. Materialized view untuk report jika perlu.
4. Hindari fetch semua data untuk All Tasks besar.

Risiko 5: Realtime membuat state frontend tidak stabil.

Mitigasi:

1. Mulai dari notification realtime.
2. Lalu comments.
3. Lalu board updates.
4. Lalu chat.
5. Gunakan invalidation/refetch sederhana sebelum optimistic update kompleks.

Risiko 6: File upload membuka celah keamanan.

Mitigasi:

1. Batasi ukuran file.
2. Validasi mime type.
3. Simpan di folder non-public jika perlu signed URL.
4. Jangan eksekusi file upload.
5. Scan file fase produksi.

Risiko 7: Performance rating disalahartikan.

Mitigasi:

1. Tampilkan formula.
2. Tampilkan data mentah.
3. Tampilkan rating sebagai indikator, bukan keputusan final.
4. Beri ruang komentar manager.

20. PENUTUP
======================================================================

Pengembangan aplikasi ini sebaiknya dilakukan bertahap dan disiplin.

Fokus utama:

1. Jangan mengorbankan stabilitas Planner + Gantt yang sudah ada.
2. Jangan membangun fitur kolaborasi tanpa auth, permission, dan audit.
3. Jangan menjadikan chat sebagai pusat semua pekerjaan; task tetap menjadi sumber kebenaran pekerjaan.
4. Gunakan data aktivitas untuk menghasilkan laporan kinerja yang objektif.
5. Bangun fitur yang paling dekat dengan kebutuhan harian user lebih dulu: My Tasks, comments, notifications, activity, performance.

Jika roadmap ini dijalankan, aplikasi akan berkembang dari MVP Project Planner menjadi platform kerja internal yang profesional, dengan kemampuan setara Cicle pada area kolaborasi, tetapi tetap punya diferensiasi kuat di Gantt, timeline, department monitoring, dan working calendar.
